## Blockers — needs fix before tag

### Blocker #1 — G2 live channel starves under slow consumer

The slow_echo backend (1 KB/s throttle on 22081) drags down the live
echo channel (22080) far below the 30% degradation budget:

- Run 1: 22080 throughput dropped from 849 KB → 153 KB (ratio 0.18)
- Run 2: 22080 throughput dropped from 538 KB → 233 KB (ratio 0.43)

This is a **real backpressure bug** — one slow channel should not be
able to consume so much of the SSH session's bandwidth/buffer that
sibling channels stall. Likely cause:
- Per-channel `MAX_WRITE_PER_CHANNEL = 16384` may be insufficient
  to prevent fast-channel starvation when one channel is in EAGAIN
  on libssh2 writes
- The round-robin in `TransportPump` may favor whichever channel
  has data first, regardless of fairness over time
- The session-level lock held during slow libssh2 writes blocks
  fast-channel progress

Investigation hint: instrument round-robin selection per channel
and EAGAIN counters in the slow_echo channel during a run.

### Blocker #2 — F→B cascade

F's `docker kill sshd` + `start sshd` sequence:

1. ESP32 detects keep-alive failure, reconnects after the 3-strike
   cascade (~90 s). `state=Connected listeners_ready=3` reported. ✅
2. Harness waits for `state=Connected` and `ch=0` then probes 22080.
3. Probe fails with `ConnectionResetError` even after 3 retries with
   2 s spacing (post-reconnect probe race fix from `931fc0e`).

ESP32 firmware reports `listeners_ready=3` but sshd's actual
`tcpip-forward` listeners are not yet operational — same Bug #2
class but post-F instead of post-network-glitch. The
`ClientAliveInterval 15` on sshd should reap the dead session within
~30 s after the kill, but apparently the new ESP32 reconnect
arrives BEFORE sshd has reaped → `Address in use` repeats, and the
new session has zero working forwards.

After F leaves the system in this state, B opens a socket, sshd
accepts the TCP, channel-open through SSH lands on a dead session
→ ESP32 never sees the bytes → 5 min of `mean=0`.

The Bug #2 fix is **not complete** — the `ClientAliveInterval 15`
mitigates network-loss scenarios but not the rapid kill+start
sequence Test F simulates (where the OLD ESP32 SSH session is
alive in the kernel right up until kill, so its socket transitions
straight to ECONNRESET — but by the time the ESP32 reconnects,
sshd may not yet have processed the FIN/RST).

Investigation hint: in F, increase the post-reconnect wait or have
the harness verify via a separate channel that sshd's listeners
are actually answering before declaring success. Or fix firmware
to send `cancel-tcpip-forward` proactively from a watchdog.

## What's already in this release that IS solid

- Bug #1 Suspect A + B: 0/15 cycles fail in 10+ runs (was 100%
  reproducible at 15/15)
- Bug #3: resolved as side-effect of Suspect B; D passes 50/50
  cycles
- Standard A test: byte-perfect at 1 MB and 10 MB across 8/8 cases
- Native tests: 40/40 PASS

## Decision

Per the release plan ("Si tout vert : git tag v2.2.0. Sinon : pas
de tag, on enquête") — **no tag pushed**. The library is
substantially better than 2.1.2 on LAN and Bug #1 is firmly closed,
but the G2 backpressure bug and the F+B cascade should be addressed
before claiming a stabilization release. Both look like one-issue
investigations rather than wholesale rewrites.

## Suggested next session

1. **G2 bug**: instrument per-channel write counters + EAGAIN
   counters across a G2 run, identify whether the issue is
   round-robin starvation or session-lock contention. Most likely
   one targeted change in `TransportPump::drainLocalToSsh` round-robin.
2. **F→B cascade**: either
   - Lengthen sshd's reap window grace before ESP32 reconnect attempt
     (firmware-side: add a brief startup grace after a detected
     keep-alive cascade before the new connect), OR
   - Have F harness wait until sshd logs "channel forwarding
     established" before considering reconnect complete (less invasive
     but Docker-log-coupled).
3. Re-run 2 consecutive full Layer 2 cycles. Tag if all green.
