# Test suite baseline report — 2026-04-28

First end-to-end run of the new two-layer test suite (host-only Unity +
integration harness against real ESP32-WROVER-KIT). This document records
what was validated, what bugs ("trous") surfaced, and what remains to
validate when the test environment cooperates.

## Hardware / environment

- ESP32 WROVER-E (8 MB PSRAM) on dev kit
- USB attachment: `/dev/ttyUSB1` from WSL2 (via usbipd-win)
- WSL2 with mirrored networking; WSL host appears at `192.168.0.186` on the LAN
- WiFi: home AP, 192.168.0.x range
- Docker Desktop on Windows, WSL integration enabled
- 3 firewall rule sets configured: Windows Defender (WSL Test SSH/Echo/SlowEcho)
  *and* Hyper-V (`{40E0AC32...}` VMCreatorId) for ports 2222 / 9000 / 9001

## Layer 2 — integration harness

### Test A — Data integrity (echo)

**Result: PASS for chunk sizes >= 8 KB. Real bug surfaced for smaller chunks.**

#### Validated end-to-end
- 1 MB transfer through `127.0.0.1:22080` (sshd container) → SSH session →
  ESP32 → echo container → back, byte-exact verification (xorshift32 PRNG)
- Throughput observed: ~30 KB/s for a single 1 MB transfer with 8 KB chunks
  on a 4-hop path (client → sshd → ESP32 → echo → return)

#### Bug found: small-chunk byte loss + intermittent BrokenPipe (HIGH priority)

**Reproduction**:
- Chunk size <= 1024 bytes, OR
- Multiple Test A cases run back-to-back in a single pytest session

**Symptoms observed**:
- `BrokenPipeError` on the first `sock.sendall()` of subsequent tests
- `recv()` returns 0 bytes (EOF) before any data arrives
- One run produced a partial transfer: 1048339/1048576 bytes (237 missing)

**ESP32 telemetry during failures**:
- `state=Connected`, no panics, no `dropped > 0` reported
- Channel slot 0 selected as inactive, then "Rejecting accepted channel
  after bind failure" appears occasionally
- Heap remains healthy (~230 KB free, 4 MB largest in PSRAM)

**Hypothesis**:
1. Channel teardown is not fully synchronous: after the previous `sock.close()`,
   the ESP32 may still be in the closing state when a new channel arrives,
   causing the next bind to be rejected (the `wait_for(ch==0)` mitigation
   helps but does not fully eliminate the race).
2. With small chunks, the `reportSendComplete` path may be losing tail bytes
   when `sock.shutdown(1)` arrives mid-flush.

**Mitigation in harness**: `A_CHUNK_SIZES` defaults to `[8 KB, 64 KB]`;
`A_CHUNK_SIZES_EXTRA = [256, 1024]` available for explicit reproduction.
`wait_tunnel_ready` now also waits for `ch == 0` before opening a new
channel.

**Where to look**:
- `src/ssh_channel.cpp` — `ChannelManager::bindChannel()` heap check + reject
  path around line 167-192
- `src/ssh_transport.cpp` — drain loop semantics when remote sends EOF
- Channel state machine `Open → Draining → Closed` transitions

### Test B — Throughput stability

**Result on phone hotspot (2026-04-29): FAIL on stability metric, but data layer healthy.**

| Metric | Observed | Threshold | Verdict |
|---|---|---|---|
| mean throughput | 44.9 KB/s | > 0 | ✅ |
| stddev/mean | 0.987 | < 0.30 | ❌ |
| stalls (2-sample windows < 1 KB/s) | 125 | 0 | ❌ |
| heap drift | 1284 B | ≤ 5 KB | ✅ |
| ESP32 `dropped` counter | 0 | 0 | ✅ |
| Symmetric counters (sent==recv) | 13.37 MB / 13.37 MB | equal | ✅ |

Interpretation: tunnel transfers data correctly (no drops, no integrity
loss, byte-perfect echo over 5 min). Stability fails because the cellular
hotspot adds high RTT variance. Re-run on stable LAN before treating
stddev/stalls as firmware bugs.

### Test D — Channel leak detection

**Result on phone hotspot (2026-04-29): FAIL on cycle-1 teardown timing,
but no leak observed.**

- Cycle 0: 0/102400 bytes (residual sshd state from Test B — see Bug #2
  in triage)
- Cycle 1: data transferred symmetrically (`sent=recv=102400`,
  `dropped=0`), but `ch` stayed at 1 longer than the 3 s threshold;
  measured ~20 s before ch returned to 0 on this run
- Heap healthy throughout

Interpretation: no leak (channel does eventually return to ch=0), but the
3 s threshold is too tight for the hotspot's latency. Either the firmware
teardown path needs tightening, or the threshold needs to be relaxed for
high-RTT environments.

### Test F — Reconnect resilience

**Result on phone hotspot (2026-04-29): PARTIAL PASS — reconnect itself
works, post-reconnect re-bind races.**

- ESP32 reconnected to sshd within 30 s after `docker kill sshd` ✅
  (snap: `state=Connected ch=0 sent=102400 recv=102400 dropped=0
  heap=226024`)
- No panic markers in serial log ✅
- Post-reconnect "hello-after-reconnect" transfer: `ConnectionResetError`
  on first `recv()` ❌

Interpretation: the firmware-level reconnect logic is sound, but sshd
takes a few seconds to re-establish the `tcpip-forward` listeners after
the ESP32's new SSH session is up. The harness waits for `state=Connected`
and `ch==0` but those are not sufficient signals — listeners-ready isn't
exposed in `STATS_TEST`. Harness improvement needed: retry the
post-reconnect probe, or add a `listeners_ready` flag in the firmware
stats.

### Test G — Circuit breaker (G1) + slow consumer (G2)

**Result on phone hotspot (2026-04-29): both FAIL, but probably due to
post-Test-F state contamination, not breaker logic.**

- G1: `breaker_trips` delta = 0 after 10 dead-port attempts (expected ≥1).
  ESP32 `sent`/`recv` counters did not budge during the test → the dead-port
  attempts never reached the firmware (see Bug #2). The breaker therefore
  had nothing to count.
- G2: `0` bytes baseline → same pattern as Test D cycle 0 (residual sshd
  state after Test F killed it).

Interpretation: needs to be re-run after a clean sshd start (or in
isolation as the first test of the session). Cannot conclude on breaker
behavior from this run.

## Bugs surfaced in 2026-04-29 run

### Bug #2 (HIGH) — Stale sshd `tcpip-forward` listeners survive ESP32 reconnect

**Reproduction**: ESP32 has live SSH session with `tcpip-forward 22080`
active. A network glitch (e.g. `Keep-alive failed: -7`) makes the ESP32
reconnect with a NEW SSH session. sshd hasn't yet detected that the OLD
session is dead, so the OLD listener on 22080 is still bound. The NEW
session's `tcpip-forward 22080` therefore fails:

```
bind [::]:22080: Address in use
channel_setup_fwd_listener_tcpip: cannot listen to port: 22080
```

From sshd's POV, the new SSH session is healthy but has zero working
forwards. The ESP32 reports `state=Connected` but `sent=0 recv=0` for
ever — clients connecting to 22080 get TCP-accept-then-EOF.

**Recovery**: `docker kill tunnel_test_sshd && docker start
tunnel_test_sshd` (NOT graceful `restart`) hard-drops all sessions
including the zombie. The ESP32 then reconnects cleanly and re-binds
all three listeners successfully. Confirmed on this run:

```
[449157] INFO  [SSH] TCP connected to 10.255.122.134:2222
[449523] INFO  [SSH] SSH handshake completed
[450176] INFO  [SSH] Reverse listener ready 127.0.0.1:22080 (bound 22080) -> 10.255.122.134:9000
[450212] INFO  [SSH] Reverse listener ready 127.0.0.1:22081 (bound 22081) -> 10.255.122.134:9001
[450382] INFO  [SSH] Reverse listener ready 127.0.0.1:22082 (bound 22082) -> 10.255.122.134:65500
[450382] INFO  [SSH] SSH session fully connected
```

After that, Test B successfully transferred 13.37 MB symmetrically.

**Where to look**:
- Firmware-side: when the keep-alive cascade fires, send an explicit
  `cancel-tcpip-forward` for each bound port BEFORE tearing the session
  down, so sshd releases the listener before the new session arrives.
- sshd-side: enable `ClientAliveInterval` so sshd notices dead clients
  faster and frees their listeners.
- Or both.

### Bug #3 (MEDIUM) — Channel teardown latency on high-RTT links

Test D shows ch returning to 0 in ~20 s on the hotspot after a
sock.close() (vs the 3 s threshold). On previous LAN runs the same
pattern returned to ch=0 in ≪ 3 s. Worth investigating whether the
firmware can fast-path EOF acknowledgements when no data is in flight.

## Environmental observations (not code bugs)

- **Hyper-V firewall rules can desynchronize** between WSL restarts and
  long-running Docker stacks. If `ESP32 timeout connecting to local
  endpoint 192.168.0.186:9000` reappears mid-session, run `Get-NetFirewallHyperVRule`
  to verify and recreate the missing one.
- **Repeated WiFi reconnection attempts trip router rate limits**: after
  several hours of test cycling we observed the AP refusing the ESP32 with
  `AUTH_EXPIRE` / `ASSOC_LEAVE` for 5-15 min. Power-cycle the ESP32 (or
  wait) to clear.
- **Initial `pio.exe` (Windows) cannot flash via `/dev/ttyUSB1`** because
  WSL's tty isn't visible to Windows binaries. Workflow used: build with
  Windows pio (faster, cached), flash with WSL `esptool` (sees ttyUSB1).
  PIO env vars must be exported via `WSLENV` to forward from WSL to
  pio.exe at build time.

```bash
  --after hard-reset write-flash -z \
  0x1000   .pio/build/test_integration/bootloader.bin \
  0x8000   .pio/build/test_integration/partitions.bin \
  0xe000   /mnt/c/Users/Denis/.platformio/packages/framework-arduinoespressif32/tools/partitions/boot_app0.bin \
  0x10000  .pio/build/test_integration/firmware.bin

# Bring docker stack up (once; do NOT down between pytest runs)

# Run a single test

python3 -m pytest -v test_a_data_integrity.py::test_echo_data_integrity[8192-1048576]

# Run all tests in a category
python3 -m pytest -v test_a_data_integrity.py -k "1048576 and not 10485760"
```
